@[card](https://zenn.dev/safubuki/articles/turtle-20260222-skillgen)

Agent Skills は、特定の作業の手順やノウハウをフォルダにまとめておき、必要な時だけAIに読み込ませる仕組みです。
チャットの内容から、いま必要なスキルだけをAIが自動で選んで発動してくれるので、指示を毎回書き直さずに済みます。ディレクトリ構造を読み込ませたり、外部ファイルを参照させたり、**スクリプト（＝プログラム）を実行させたり**することもできます。この基本的な仕組みや利点は前回の記事で詳しく書いたので、そちらも参照してください。

前回はこのAgent Skillsの素晴らしさを紹介しました。使えるスキルが増えるほど作業は楽になります。
ただ、自分でスキルを量産していると、次のような悩みにも気づきます。

- 指示は `SKILL.md` に自然言語で書くので、ちょっとした作業でも意外とトークンやクレジットを消費する
- 毎回同じ作業を頼みたいのに、内部でその都度プログラムを生成し直すので、時間とクレジットの無駄になる
- スクリプトも同梱できるはずなのに、意外とスクリプトまでは作ってくれない
- そもそも、どこを定型作業として切り出すべきかの判断も難しい

かといって、定型作業をさせたいスキルを作るたびに「ここはスクリプトにして」と毎回指示するのも面倒です。

![たこ亀](/images/turtle-20260705-scripted-skillgen/たこ亀.jpeg)

そこで作ったのが `scripted-skill-generator` です。ひとことで言うと、**定型作業はスクリプトに任せ、判断が必要なところはAIが受け持つ。そんな役割分担を最初から組み込んだスキルを作ってくれるスキル** です。全部をスクリプトにするのではなく、決まった処理だけをスクリプトへ、その場で考えることはAIへ、と振り分けてくれます。

## 定型処理はスクリプト、判断はAI

`scripted-skill-generator` の考え方はシンプルです。
**まず「毎回同じ結果でよい定型作業」がないかを洗い出し、その部分はスクリプト（プログラム）で自動化する。** そして、状況によって答えが変わる、人の判断がいるところだけをAIに任せます。

| 処理 | AIだけに任せると起きること | スクリプトに任せる効果 |
| --- | --- | --- |
| GitHubの活動履歴を取得する | 取得する項目やまとめ方が毎回少しずつ変わる | 取得する項目と出力の形を毎回同じにできる |
| データを集計する | 集計の仕方や並び順が毎回変わる | 並び順や文字コードを毎回同じにできる |
| プロジェクト内のルール違反を探す | 本来は機械的に処理できるところまでAIに判断させることになる | ルールをコードに残し、毎回同じ基準で処理できる |
| データを表に変換する | 量が多いとその読み込みだけでトークンを使う | 変換処理だけを速く終わらせられる |

このように「入力と出力の形が決まっている作業」は、AIに毎回考えさせるより、スクリプトに任せたほうが結果が安定します。
一方で、仕様がまだ固まっていない設計、レビューでどこから直すかの優先順位づけ、相手に合わせて変わる文章づくりなどは、AIに残したほうが自然です。
AIを使わなくする話ではなく、**決まりきった作業はスクリプトに任せて、AIには判断に集中してもらう**という役割分担です。決まった作業とその場の判断を分けて考えるのが、この記事でいちばん持ち帰ってほしい考え方です。

## 前回のスキルからどう変わったか

前回の `skills-generator` は、Agent Skill全般を作る・改善する・診断する、幅広いスキルでした。
`scripted-skill-generator` は、その良さをそのままに、**「定型作業をスクリプトに逃がす」という視点を最初から組み込んで進化させたもの**です。まったく別物というより、`skills-generator` の続きだと思ってもらえれば近いです。

具体的には、スキルを作るときに次のような段取りを踏んでくれます。

- 最初に「どこまでスクリプトにするか」と「使う言語（PythonかNode.jsか）」を選ばせてから作り始める
- でき上がるのは、`SKILL.md`（AIへの手順書）＋ `scripts/`（実行するプログラム）＋ 入出力の約束ごとを書いたメモ、の3点セット
- ファイルを作るときは、いきなり書き込まず「これから何を作るか」を先に見せてくれる
- 作ったスキルの形がおかしくないか、あとから機械的にチェックできる

以降で、この中身を順に見ていきます。

## `scripted-skill-generator` の中身

フォルダ構成は、役割ごとに分かれています。

```text
.agents/skills/scripted-skill-generator/
├── SKILL.md
├── assets/
│   └── scripted-skill-template.md
├── references/
│   ├── runtime-policy.md           # どの言語で書くか、環境をどう扱うかの方針
│   ├── script-first-design.md      # 何をスクリプトにし、何をAIに残すか
│   └── script-quality-checklist.md # 作ったスクリプトが安全か確かめる観点
└── scripts/
    └── scripted_skill_tool.py      # スキルの雛形づくりを助ける補助プログラム
```

スクリプト活用型らしいのは、`references`（判断のよりどころを書いたメモ）の中身と `scripts/` の存在です。
ここでは、その要になる3ファイルを見ていきます。

### script-first-design.md — スクリプトにする線引き

「何をスクリプトにして、何をAIに残すか」を判断するためのメモです。
スクリプトにした方がいい作業の条件が、次のように整理されています。

| こういう作業は… | たとえば |
| --- | --- |
| 入力と出力の形が決まっている | データを集計して表に出す |
| データの量が多い | 数千件の履歴をまとめる |
| 同じ処理を何度も繰り返す | 毎月のレポート集計 |
| きっちり正確に読み取りたい | 日付、数値、ログの処理 |
| プロジェクト独自のルールがある | 命名の決まり、ファイルの置き場所 |

逆に、要件を固める作業、設計の判断、レビュー、優先順位づけ、一度きりの小さな作業は、AIに任せたままにします。
スクリプトにするのは、AIを使わないためではなく、**AIに毎回同じ作業を一から考え直させないため**です。

### runtime-policy.md — 勝手に環境を変えない

作られるスクリプトは、WindowsでもLinuxでも動くことを前提にしています。
使う言語は、まずPython、次にNode.js、それでも足りなければPowerShellとBash、という優先順位です。特別な事情がなければ、どのPCにも入りやすいPythonが第一候補になります。

ここでこだわっているのが、**必要な言語が入っていなくても、勝手にインストールしない**ことです。
AIが気を利かせて `winget` や `apt`（ソフトを自動で入れるコマンド）を走らせ、いつの間にか自分のPCの環境が変わっていた、というのは避けたい。
なので、言語が見つからないときは「インストールする」「別の言語に切り替える」「今回はファイルを作るだけにする」を、ユーザーに選んでもらう作りにしています。実際にスクリプトの中身を読んでも、無断でインストールするコードは入っていません。

### script-quality-checklist.md — 生成物が安全に動くように

このスキルの生成物には「実際に動くプログラム」が含まれます。ただ文章を書くスキルとは違って、動く以上は事故が起きないよう、私なりに気をつけているルールをここに集めています。たとえば、こんなことです。

- ファイルを書き換えたり消したりする前に、まず「何をするか」を見せてから実行する
- ファイルの場所の指定は、WindowsとLinuxの違いでつまずかない書き方にする
- パスワードやトークンのような秘密の情報を、画面やログに出さない
- 出力の並び順をいつも同じにして、実行するたびに結果がブレないようにする

## 前回のスキルと同じ依頼で比べてみる

使い方は前回と同じで、GitHubで公開しているリポジトリをクローンして、`scripted-skill-generator` を自分の環境に置くだけです（配置場所や設定は前回の記事と同じです）。あとは、作りたいスキルをAIに頼めば動き出します。

@[card](https://github.com/safubuki/scripted-skill-generator)

違いがいちばん分かりやすいのは、前回と同じものを作らせてみることです。
前回の記事では、`skills-generator` で「フロントエンド向けの README 生成スキル」を作りました。`package.json` を読んで、依存ライブラリ一覧やセットアップ手順を自動生成するスキルです。

そこで、`skills-generator` と `scripted-skill-generator` に、**まったく同じ依頼**を出して作らせてみました。

```text:どちらにも出した依頼
フロントエンドプロジェクト向けの README 生成スキルを作って。
package.json の依存関係を読み込み、セットアップ手順・依存ライブラリ一覧を
自動生成する実用的なドキュメントスキルにしてほしい。
```

でき上がったスキルの構成が、はっきり分かれました。

```text
【skills-generator で作った場合】        【scripted-skill-generator で作った場合】
readme-generator/                        readme-generator/
├── SKILL.md                             ├── SKILL.md
├── assets/                              ├── scripts/
│   └── readme-template.md               │   └── extract_package_meta.py  ← 実行できるプログラム
└── references/                          └── references/
    └── package-json-fields.md               └── script-contract.md
```

左の `skills-generator` 版は、`package.json` の読み方や依存ライブラリの分類ルールを、**すべて `SKILL.md` の中に文章で書いています**。スクリプトはありません。つまり README を作るたびに、AIがその手順を読んで解析をやり直します。

右の `scripted-skill-generator` 版は、そのうち **「`package.json` から事実を取り出す」部分をプログラム（`extract_package_meta.py`）に切り出しました**。実際に動かすと、依存ライブラリ、開発用ライブラリ、スクリプト、必要なNodeのバージョン、パッケージマネージャ（npm / pnpm / yarn）、使われているフレームワークを、決まった形で並べて返します。

```text:スクリプトの出力（一部）
"install_command": "pnpm install",
"frameworks": ["Next.js", "React"],
"node_version": { "engines": ">=18.17.0" }
```

大事なのは、**これで README 作りからAIが消えるわけではない**ことです。
スクリプトが受け持つのは「`package.json` に書いてある事実を、正確に、毎回同じ形で取り出す」ところまで。そのフレームワークがこのプロジェクトで何をしているかの説明、構成の良し悪し、README全体の文章のトーンは、AIが引き続き担当します。実際、右のスキルの `SKILL.md` には「事実はスクリプトの出力を根拠にし、依存名やバージョンを記憶で書かない」と明記されていました。

**変わらない事実の取り出しはプログラム、読み手に合わせた説明はAI。** 同じ依頼でも、この線引きが最初から組み込まれるのが `scripted-skill-generator` の違いです。

## ほかにもこんな作業に向いている

READMEの例で形が見えると、「あの作業もスクリプトに任せられそう」というものが出てきます。私なら、次のような「決まった処理」と「判断」がきれいに分かれる作業から、こういうスキルにしていきます。

| 作業 | スクリプトに任せる部分 | AIに任せる部分 |
| --- | --- | --- |
| GitHub活動レポート | データ取得、期間・種別ごとの集計、表への書き出し | 成果の見方、気になる数字の解釈、レポート本文、改善案 |
| ブログ記事チェック | タイトルやトピックの確認、画像リンク切れの検出、避けたい表現の洗い出し | 構成の良し悪し、読者に刺さる言い回しへの調整、事実確認 |
| プロジェクトのルールチェック | 対象ファイルの走査、決めたルール違反の検出、場所と理由の一覧化 | 直し方の提案、既存の作りとの整合、影響範囲の説明 |

どれも、左側は「毎回同じでいい作業」なのでスクリプトに任せ、右側の「その時々で変わる判断」だけAIに残す、という同じ考え方です。

## 安全に使うための工夫

スクリプト、つまり実際に動くプログラムを同梱できるということは、便利さと同じだけ気をつけたいこともあります。ファイルを大量に消したり、外部にデータを送ったり、`.env` のような秘密の情報を読んだり、といった動きは特に注意したいところです。

前回の記事でも「野良スキルのセキュリティリスク」に触れましたが、自分で作るスキルなら、その注意点を最初から作り込んでおけます。`scripted-skill-generator` では、次の3つを基本にしています。

1. ファイルを作る・書き換えるときは、いきなり実行せず「これから何をするか」を先に見せる
2. 必要な言語が入っていなくても、勝手にインストールしない
3. でき上がったスキルの形がおかしくないか、あとから機械的にチェックできる

もちろん、これだけで安全がすべて保証できるわけではありません。それでも、スキルを作るたびに同じ観点を通す入口があるだけで、だいぶ安心して手を動かせます。

## まとめ

前回の「スキルを作るスキル」に続き、今回はその発展形として、定型作業をスクリプトに逃がす `scripted-skill-generator` を紹介しました。

大事なのは、**決まりきった処理はスクリプトへ、その場の判断はAIへ**という役割分担です。同じ「READMEを作って」という依頼でも、この分担が最初から組み込まれる。決まった処理は毎回同じ結果で速く、判断はこれまで通りAIが担う。両方のいいとこ取りができるのがうれしいところです。

また、トークンの節約にもつながると考えています。スクリプトがあれば、スキルは決まったプログラムを呼び出すだけ。以前のように、実行のたびにAIがスクリプトを組み立て直し、「どんなスクリプトにするか」まで毎回考える、という手間がなくなります。どれくらい効くかは、これから実際に測って確かめたいところです。

`scripted-skill-generator` は、`skills-generator` の良さはそのままに、スクリプトによる定型化をより強化して進化したものです。私は今後 `scripted-skill-generator` をメインに使うと思います。まずはこれまで作ってきたスキルを見直して、毎回AIに任せていた定型作業を少しずつスクリプトに置き換えていくつもりです。同じ結果で良い作業ほど、スクリプトにした恩恵は増えていくはずです。
